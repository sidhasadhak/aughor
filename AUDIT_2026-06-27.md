# Aughor — Full Repository Audit (2026-06-27)

## Executive Summary

Aughor is an ambitious, single-author **autonomous data-warehouse intelligence platform** (~72k LOC Python, ~32k LOC TS) that differentiates on *deterministic trust guards* over LLM-generated SQL — its biggest risk is **one security bypass that lets the Query Builder execute arbitrary mutating SQL**, plus a growing surface of dead code and a breached test-ratchet that signals discipline is slipping; its biggest opportunity is to **delete the 30% of the codebase that isn't on the live path** and convert the impressive guard infrastructure into the product's external, verifiable trust story.

## Architecture Overview

**Stack:** FastAPI + LangGraph backend (Python 3.11), Next.js 16 / React 19 / Tailwind frontend, DuckDB/Postgres/etc. connectors, SQLite for registry/history/ledger/audit, Qdrant for vectors, Fernet for secrets at rest. Single-process, runs fully local via Ollama.

**Request flow:**
- `aughor/api.py:51` — lifespan boot: kernel journal → samples → org/workspace migration → metastore sync → connection validation → explorer resume → ontology refresh loop → schedulers. Each step fault-isolated, non-fatal.
- **Three answer modes** share one SQL-safety pipeline (`aughor/sql/safety.py:31` `preflight_repair`): *Insight* (chat NL2SQL), *Deep* (ADA investigative loop, `routers/investigations.py`), *Explorer* (background autonomous learning, `explorer/agent.py`).
- **Job Kernel** (`aughor/kernel/jobs.py`) supervises all long work: state machine + heartbeats + boot recovery + idempotency + scope cancellation. Replaces 17 raw `create_task` sites.
- **Connection layer** (`aughor/db/`): registry (Fernet-encrypted DSNs), pool (exclusive checkout, idle TTL, health check), `connection.py:47` `_security_pre` (SafetyChecker + AST `is_mutating`) and `_security_post` (PII redaction, audit, row budget).
- **Licensing** (`aughor/licensing/`): additive Free/Pro/Enterprise tiers, defaults to Enterprise (dark), 402s with upgrade hint, opt-in per route via `gate(Capability)`.

**Intent & values (from README + commits + code shape):** Not a SQL copilot — an *always-thinking* agent that builds a business ontology, explores continuously, and is engineered so "the numbers are trustworthy, not just plausible." Values, in priority order: **trustworthiness > autonomy > developer-ergonomics-as-discipline (test ratchets) > local/privacy > breadth of connectors.** The implicit "definition of done" is: every guard ships with a test that proves it fires, no silent except-pass (must use `tolerate()`), graceful-by-contract (never a 500/hang/silent-wrong-success).

**Confidence:** High on architecture and the security finding (verified in code). Medium on UX specifics (didn't run the frontend end-to-end). Guessing that no multi-user auth is intentional (single-user local tool).

The `_security_pre`/`_security_post` split is the right architecture — it's specifically the *bypass* of it that's the critical bug.

## The 10x Roadmap (Phase 4)

The project's core value is **"numbers you can act on"** — deterministic trust over LLM prose. The 10x move is to make that trust *externally verifiable*, not just internally enforced.

### The single highest-leverage change
**Fix the Query Builder bypass (#1) and make the Trust Receipt the product's public contract.** Right now the most interactive user surface (Query Builder) is the *one place* that skips every guard the project built its identity around. Closing it is ~20 lines + 1 test, and it converts the trust substrate from "we assert it" to "we can prove it on every answer, including the user-typed SQL path." That single fix realigns reality with the README's central claim.

### Sequenced recommendations

**Quick wins (hours–days, high impact/effort):**
1. **Fix #1** (Query Builder/bulk_read safety bypass) — the security hole. ~1 day incl. regression test. *Impact: closes the only path to a destructive wrong-number/wrong-data outcome.*
2. **Fix #3** (restore the test ratchet) — convert 34 new silent swallows to `tolerate()`, bump baseline. ~half day. *Impact: re-enables the discipline mechanism that's been the project's quality engine.*
3. **Delete dead code (#6, #8)** — `aughor-v2/charts/`, `_ConnectionsScreen_DEPRECATED`, rename `useLearnedSkill`. ~2 hours. *Impact: eslint returns to `--max-warnings=0`, CI can gate, -1.7k LOC of confusion.*
4. **Fix #2/#9** (info leak + upload cap) — ~2 hours. *Impact: closes two more info-disclosure / DoS edges.*

**Structural bets (weeks, the actual 10x):**
5. **Make the Trust Receipt queryable and public.** The `Trust Receipt` / `Evidence Ledger` already exists internally (`investigations.py:93` `_write_answer_receipt`, `kernel/ledger.py`). Expose `GET /receipt/{id}` returning the signed lineage (executed SQLs, input tables, guards that fired, earned-confidence) for *every* answer — chat, query-builder, monitor — and render it inline in the UI as a one-click "why this number." This is the latent potential the authors have built but not exploited: the guard infrastructure is far ahead of its surface area. *Effort: ~1 week. Impact: this is the 10x — it turns "trustworthy by construction" into "trustworthy by inspection," which is the only form competitors can't copy with a prompt.*
6. **Bound the system (#4 + #5).** Global job semaphore, scheduled `mat_cache` eviction, APScheduler-driven `evict_expired`. The "runs fully local on a laptop" promise dies the moment a user opens three tabs. *Effort: ~3 days. Impact: durability.*
7. **Compress the docs/FEATURES surface.** `FEATURES.md` is 3,829 lines and `ROADMAP.md` 345 — for an alpha with one contributor. The documentation mass is itself a maintenance liability and signals scope creep over depth. Consolidate to one living FEATURES.md ≤400 lines and move the rest to `docs/archive/`. *Effort: ~1 day. Impact: the project tells a tighter story to itself and to users.*

**What to delete entirely:**
- `web/aughor-v2/charts/` (abandoned parallel rewrite, only theme CSS used).
- `_ConnectionsScreen_DEPRECATED` and the entire deprecated-screen pattern in `page.tsx` (one 2,341-line file is itself a smell — but killing the dead functions is the subtraction that matters first).
- `AUDIT_REPORT.md` from 2026-05-29 — superseded; keep one current audit doc, not a sediment of them.
- The 10 stale local branches (`2026-06-13-*`) — prune.

## Open Questions (couldn't verify without running it)

1. **Does the explorer's manifest-driven mode actually stay within budget on a real warehouse?** The README claims ~17× fewer tokens; I'd run `evals/run_realdb` against a 10+ table Postgres to confirm the token-ratchet, not just the unit-level coverage test.
2. **The ontology refresh loop** (`api.py:369`) calls `db.get_schema()` then closes — I couldn't confirm whether `invalidate_ontology` + a bare `get_schema()` actually triggers a rebuild, or just a cache drop that waits for the next read. Needs a live run with `AUGHOR_LOG_LEVEL=DEBUG`.
3. **Qdrant dependency** — `pyproject.toml` lists `qdrant-client` as core (not optional), but the README implies local-only. Is a running Qdrant server required for boot, or does the vector store degrade gracefully? The `_setup_samples` path and several `scroll_payloads` calls suggest it's on the live path; if so, "runs fully local" needs an asterisk. I'd run `./start.sh --api-only` with Qdrant stopped and watch `/health`.
4. **The 300s LLM read timeout** (`provider.py:235`) — is there a corresponding *cancellation* path when a client disconnects mid-SSE? The kernel cancels the job, but I didn't trace whether the in-flight `httpx` call is aborted or just left to time out.
